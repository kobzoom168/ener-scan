# Moldavite — รายงาน HTML ฉบับ V2 (System Design ต่อเนื่อง)

**สถานะเอกสาร:** ร่างสเปกระบบ (continuation) · อ้างอิงทิศทาง Ener Scan V2 และแยก lane Moldavite ชัดเจน  
**ขอบเขต:** เฉพาะ **เส้นทางรายงาน HTML แบบ Moldavite รุ่นใหม่ (V2)** — ไม่แทนที่การออกแบบ HTML แบบ legacy ทั้งระบบ  
**ไม่ครอบคลุม:** Flex summary-first Moldavite (คงหลักการ “สั้น / handoff” เดิม) · การตรวจจับ Moldavite · routing Gemini

---

เอกสารนี้เป็นส่วน **ต่อจาก** แนวคิด system design ที่กำหนดว่า ReportPayload แยก slice `moldaviteV1` และ HTML public report สามารถลงลึกกว่า Flex ได้ โดยยังคง semantic lane เดียวกับ Moldavite (หินเทคไทต์ / โทนเร่งการเปลี่ยนแปลง) และไม่ดึงภาษาหรือโครงสร้างของพระเครื่องไทยหรือคริสตัลเจเนอริกทั้งระบบ

---

## 1. วัตถุประสงค์ของ Moldavite HTML Report V2

### 1.1 ทำไมต้องเป็นเส้นทางรายงานแยก

- **Moldavite มี “ภาษาเฉพาะ lane”** — โฟกัสการเปลี่ยนแปลง การปล่อยของเก่า และจังหวะเริ่มรอบใหม่ ไม่ใช่พลังคุ้มครองแบบพระเครื่องหรือ meta แบบคริสตัลเจเนอริกทั่วไป  
- **ผู้ใช้ที่เข้ามาจาก Flex คาดหวังความต่อเนื่องของความหมาย** — HTML V2 ต้อง “ต่อยอด” จากการ์ดสั้น ไม่ใช่เปลี่ยนโลกอธิบายไปเป็น generic report  
- **รูปแบบการเล่าเรื่องที่ต่างจาก legacy** — legacy เน้นข้อความยาว หลายการ์ด และบางส่วนอิง energy family ทั่วไป; Moldavite V2 ตั้งใจให้ **กราฟเป็นตัวเล่าเรื่องหลัก** และข้อความเป็นเลเยอร์รอง

### 1.2 ทำไมต้องไม่ mutate รายงาน HTML แบบ legacy

- **ความเสถียรของผลิตภัณฑ์** — ผู้ใช้และทีมปฏิบัติการอาจอ้างอิง layout / ข้อความ legacy เป็น baseline  
- **ความเสี่ยงด้าน regression** — การผสาน layout ใหม่เข้า template เดียวกับทุกคริสตัลจะเพิ่มความซับซ้อนและการทดสอบ  
- **แยก versioning ชัด** — Moldavite V2 ควรระบุได้ว่าเป็น `renderer` หรือ `presentationId` แยก (เช่น `moldavite-html-v2`) เพื่อ rollback / เปิดเฉพาะ cohort ได้

### 1.3 ทำไมทิศทางใหม่คือ graph-first

- **ลด cognitive load** — ผู้ใช้จำภาพรวมจากกราฟได้เร็วกว่าอ่านย่อหน้ายาว  
- **สื่อ “โปรไฟล์เจ้าของ” กับ “โปรไฟล์หิน” พร้อมกัน** — radar / overlay สองชั้น บอกเรื่อง **ความสอดคล้องและความตึง** ได้ชัดกว่าข้อความล้วน  
- **สอดคล้องกับข้อมูลที่มีอยู่แล้วใน lane** — คะแนนมิติชีวิต (งาน / ความสัมพันธ์ / การเงิน) และวันเกิดสำหรับโปรไฟล์เจ้าของ นำไปสู่การแสดงผลเชิงปริมาณได้โดยไม่ต้องเพิ่มเรื่องเล่าแบบ self-help ยาว

## 2. หลักการออกแบบ (Design principles)

| หลักการ | คำอธิบาย |
|---------|-----------|
| **รักษาเอกลักษณ์ Moldavite** | โทนการอ่าน = การเปลี่ยนแปลง / tektite / โทนเขียว (ตามระดับ confidence ในการตั้งชื่อ subtype) — ไม่สลับไป meta พระเครื่องหรือ generic crystal |
| **พลังหลักคงที่** | แกนเชิงความหมายของพลังหลักยังคง **`เร่งการเปลี่ยนแปลง`** เป็น default lane (ยกเว้นการตั้งชื่อ confidence ที่ไม่ใช้คำว่ามอลดาไวต์โดยตรง — ยังอยู่ใน lane เดียวกัน) |
| **ลด prose ยาว** | แต่ละส่วนมีงบประมาณความยาว (เช่น บรรทัดเดียว–สามบรรทัด) — รายละเอียดเชิงลึกให้กราฟ + คำบรรยายสั้นใต้กราฟ |
| **กราฟเป็นคำอธิบายหลัก** | ผู้ใช้ควร “เข้าใจภาพรวม” จากกราฟภายใน ~10 วินาที ก่อนอ่านข้อความ |
| **โปรไฟล์เจ้าของจากวันเกิด** | ใช้ DOB เป็นแหล่ง derive โปรไฟล์เชิงแกน (ไม่ใช่คำทำนายชะตาแบบโหราศาสตร์เต็มระบบใน MVP — แต่เป็น **โปรไฟล์บุคลิก/จังหวะ** ที่ map ไปแกนเดียวกับกราฟหิน) |
| **ชื่อ subtype ตาม confidence** | เกณฑ์การแสดง “มอลดาไวต์” vs “หิน/คริสตัลโทนเขียว” คงตามระบบ naming เดิม — HTML V2 แสดงสอดคล้องกับ Flex |

## 4. สเปก Radar graph

### 4.1 แกน (axes) — ข้อเสนอระดับสเปก

เพื่อให้สอดคล้องกับผลิตภัณฑ์ปัจจุบัน (มิติชีวิตสามด้าน) และยังเล่าเรื่อง “การเปลี่ยนแปลง” ได้ แนะนำ **แกนหลัก 3 แกน** ดังนี้ (ชื่อแสดงผลเป็นภาษาไทย):

1. **งาน** — โครงสร้าง เป้าหมาย บทบาท การเริ่ม/ปรับกรอบใหม่  
2. **ความสัมพันธ์** — ขอบเขต การสื่อสาร การเคลียร์ดีลทางใจหรือคำพูด  
3. **การเงิน** — กระแส การตัดสินใจเชิงโครงสร้าง การปรับกลไก (ไม่ใช่คำแนะนำลงทุน)

**เหตุผลที่ใช้สามแกนนี้** — สอดคล้องกับ life-area ที่ระบบคำนวณแล้ว และผู้ใช้เข้าใจง่ายเมื่อเทียบกับ Flex

**แกนเสริม (Phase 2+)** — หากต้องการความละเอียดบุคลิกเจ้าของเพิ่ม อาจเพิ่ม 1–2 แกนเชิง “จังหวะ” (เช่น การเปิดรับความไม่แน่นอน / การยึดแผนเดิม) แต่ต้องระวัดระวังไม่ให้แกนเยอะเกินอ่านในมือถือ

### 4.2 Owner layer

- แสดงเป็น **polygon ชั้นที่หนึ่ง** (เส้น/พื้นโปร่ง)  
- ค่า derive จาก **วันเกิด** ผ่านฟังก์ชัน mapping ภายในระบบ (เชิงสัญลักษณ์/บุคลิกภาพ ไม่ใช่คำทำนายเชิงโหราเต็มรูปแบบใน MVP)  
- ความหมาย: “แนวโน้มพื้นฐานของเจ้าของบนแกนนี้”

### 4.3 Crystal layer (Moldavite)

- แสดงเป็น **polygon ชั้นที่สอง**  
- ค่า map จาก **คะแนนมิติชีวิต Moldavite** (และ/หรือ normalization เดียวกับ owner เพื่อให้อยู่สเกลเดียวกัน)  
- ความหมาย: “พลังโทนเปลี่ยนแปลงของชิ้นนี้ไปเน้นที่มิติใดเมื่อเทียบสเกลเดียวกัน”

### 4.4 การอ่าน alignment / tension

- **Alignment (สอดคล้อง)** — แกนที่ทั้งสองชั้นใกล้กัน หรือทิศทางเสริมกัน (นิยามเป็นกฎใน logic layer เช่น ระยะห่างค่าต่ำกว่าเกณฑ์)  
- **Tension (ตึง)** — แกนที่ห่างกันมาก หมายถึง “จุดที่ต้องใช้สติ / ลดจังหวะ / ไม่เร่งเกิน” — **ไม่ใช่คำว่าเลวร้าย** แต่เป็น “จุดที่ไม่สบายตัวอาจเกิดได้”

### 4.5 สิ่งที่ผู้ใช้ควร “พกกลับไป” จากกราฟ

- มิติไหนที่ **เสริมกัน** — ใช้ชีวิตในช่วงนั้นอาจรู้สึก “ลื่น” กับโทนหิน  
- มิติไหนที่ **ตึง** — ไม่ได้แปลว่าห้ามทำ แปลว่า **ต้องจัดจังหวะและสังเกตความรู้สึก**  
- ภาพรวมเดียวที่จำได้: **สองชั้นบนแกนเดียวกัน**

## 5. โมเดลข้อมูล (Moldavite HTML V2) — โครงสร้างฟิลด์

> ระดับนามธรรมสำหรับ implementation ภายหลัง — ชื่อฟิลด์อาจ map เป็น JSON ใน `moldaviteV1` หรือ slice ย่อย `moldaviteHtmlV2`

### 5.1 `hero`

- `subtypeLabel` — ชื่อที่แสดง (ตาม confidence naming)  
- `tagline`  
- `mainEnergyLabel` — ค่า default lane  
- `openingHook` — 1–2 ประโยค  
- `objectImageUrl`, `reportGeneratedAt`

### 5.2 `graph`

- `axes[]` — `{ id, labelThai, order }`  
- `ownerSeries[]` — `{ axisId, value0to100 }`  
- `crystalSeries[]` — `{ axisId, value0to100 }`  
- `scale` — คำอธิบายสเกล (เช่น 0–100)  
- `legend` — owner vs crystal  
- `alignment` — `{ axisId | null, summaryPhrase }`  
- `tension` — `{ axisId | null, summaryPhrase }`

### 5.3 `ownerProfile`

- `summaryLine` — หนึ่งบรรทัด  
- `traits[]` — สั้น ๆ 2–4 ข้อ  
- `derivationNote` — ข้อความโปร่งใสว่ามาจาก DOB แบบใด (ไม่แสดงสูตรดิบถ้าไม่จำเป็น)

### 5.4 `interactionSummary` (How the stone works with you)

- `headline`  
- `bullets[]` — 2–4 ข้อ  
- `alignmentEcho` — อ้างอิงสั้น ๆ จากกราฟ (ไม่ซ้ำยาว)

### 5.5 `lifeAreaDetail[]`

- รายการต่อมิติ: `{ key: work | relationship | money, score, shortText }`  
- เรียงตามคะแนน สูง → ต่ำ

### 5.6 `usageCaution`

- `bullets[]`  
- `disclaimer` — ข้อความมาตรฐานระบบ

## 6. Logic mapping (แนวคิด ไม่ใช่โค้ด)

### 6.1 DOB → owner profile axes

- ใช้วันเกิดเป็นอินพุตเข้า **ฟังก์ชัน derive แบบกำหนด** (deterministic หรือตาราง lookup) เพื่อได้ค่า 0–100 ต่อแกน  
- ต้อง **ป้องกันการอ่านเป็นโหราศาสตร์เต็มรูปแบบ** ใน copy — เน้นว่าเป็น “แนวโน้มบุคลิก/จังหวะ” บนแกนที่ออกแบบไว้

### 6.2 Moldavite life-area scores → crystal layer

- คะแนนงาน / ความสัมพันธ์ / การเงิน ถูก **normalize** ให้อยู่สเกลเดียวกับ owner layer บนแกนเดียวกัน  
- หากมีการ map เพิ่มจากแหล่งอื่นในอนาคต ต้องไม่ทำลายความหมาย “มิติชีวิตสามด้าน”

### 6.3 Strongest alignment

- เลือกแกนที่ **ระยะห่างระหว่าง owner กับ crystal น้อยที่สุด** (หรือกฎ weighted ที่ระบุในสเปก implementation)  
- สรุปเป็นประโยคสั้นใน `graph.summary` และอาจสะท้อนใน `interactionSummary`

### 6.4 Strongest tension

- เลือกแกนที่ **ระยะห่างมากที่สุด**  
- Copy ต้องเป็นเชิง “ระวังจังหวะ / สังเกตความรู้สึก” ไม่ใช่คำตัดสินชีวิต

## 7. Copy strategy

- **ข้อความสั้นเป็นค่าเริ่มต้น** — งบประมาณความยาวต่อการ์ดชัดเจนใน QA  
- **กราฟเล่าเรื่องหลัก** — ข้อความเป็น “คำบรรยายใต้ภาพ”  
- **หลีกเลี่ยง** — เรื่องเล่นแรงแบบ self-help, การันตีโชคลาภ, ศัพท์พระเครื่อง, การผสาน wording คริสตัลเจเนอริกทั้งระบบ  
- **รักษา lane Moldavite** — ทุกประโยคต้องอ่านแล้วรู้ว่าเป็น “โทนเปลี่ยนแปลง / tektite lane” ไม่ใช่พลังเมตตาหรือบุญบารมีแบบไทย

---

## 8. ทิศทาง UI

- **Graph-first** — พื้นที่กราฟใหญ่สุดในเนื้อหาหลักหลัง Hero  
- **Premium** — โทนสีเข้ม คอนทราสต์ดี เงา/เส้นบาง ไม่รก  
- **Whitespace** — ระยะหายใจระหว่างการ์ดชัด  
- **สีเน้นเขียว Moldavite** — ใช้เป็น accent เดียวที่จำแนก lane นี้จาก legacy ทอง  
- **ลำดับชั้นภาพ** — กราฟ > สรุปสั้น > รายละเอียดมิติ > disclaimer  
- **ข้อความเป็นเลเยอร์รอง** — ไม่แย่งจุดโฟกัสจากกราฟ

---

## 9. ลำดับการทำ MVP (แบ่ง Phase)

### Phase 1 — โครงและความจริงขั้นต่ำ

- แยก renderer / template ชัดว่าเป็น Moldavite HTML V2 (ไม่ทับ legacy)  
- แสดง Hero + **radar แบบ static** (ข้อมูลจากคะแนนมิติที่มี) + graph summary สั้น  
- Owner profile แบบ **placeholder โปร่งใส** หรือ derive แบบง่ายจาก DOB (ถ้ายังไม่พร้อม ให้แสดงข้อความว่า “เวอร์ชันถัดไป”) — **เลือกอย่างใดอย่างหนึ่งตามทรัพยากร**  
- life-area detail + usage/caution แบบสั้น

### Phase 2 — Owner layer เต็มรูปแบบ

- DOB → owner series ครบทุกแกน + copy โปรไฟล์เจ้าของที่ผ่านเกณฑ์ภาษา  
- ปรับปรุงกฎ alignment/tension และการทดสอบกับผู้ใช้จริงแบบจำกัด

### Phase 3 — ความลื่นและคุณภาพ

- micro-interaction (hover/legend บนเว็บที่รองรับ)  
- การปรับ contrast / responsive บนจอเล็ก  
- การวัด success metrics (เวลาอยู่หน้า, scroll depth, อัตรากลับมาเปิดรายงาน)

## 10. เกณฑ์ความสำเร็จ (Success criteria)

- **ความต่อเนื่องของความหมาย** — ผู้ใช้ที่มาจาก Flex Moldavite รู้สึกว่า HTML เป็น “ฉบับเต็มของเรื่องเดียวกัน” ไม่ใช่รายงานคนละโลก  
- **ความเข้าใจเร็ว** — ผู้ใช้ส่วนใหญ่อธิบายได้ด้วยตนเองว่า “หินไปแรงที่มิติไหน” และ “ตัวเองไปแรงที่มิติไหน” ภายหลังดูกราฟสั้น ๆ  
- **ความสั้นของข้อความ** — ไม่มีการ์ดใดยาวเกินงบประมาณที่กำหนดโดยไม่ตั้งใจ  
- **ความปลอดภัยของภาษา** — ไม่มีข้อร้องเรียนเชิง “สัญญาผลลัพธ์” หรือสับสนกับพระเครื่อง  
- **ความแยกทางเทคนิค** — legacy report ยังเรนเดอร์และทดสอบได้เหมือนเดิม; Moldavite V2 เปิด/ปิดได้อิสระ
